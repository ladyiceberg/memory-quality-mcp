# Memory Quality MCP · Benchmark 系统评估报告

> 版本：v1.0 · 评估日期：2026-04-05
> 数据规模：500 条 scenario，833 条 memory，200 条有效评分

---

## 一、执行摘要

本次 Benchmark 覆盖 Memory Quality MCP 完整三环链路（数据筛选 → 记忆提取 → 评分引擎），总体结论如下：

**记忆提取环节基本可用**，73.6% 的 scenario 能产出有效 memory，JSON 解析失败率 6%，处于可接受范围。**评分引擎存在结构性缺陷**：对"应该删除"的临时性内容，误留率高达 78%（delete 类精确率仅 22%），是整条链路最薄弱的环节。**LLM 稳定性风险不可忽视**：21% 的评分失败被强制标为 3.0 分复查，污染了准确率统计的可信度。综合判断，当前系统**不建议在生产环境全自动执行删除操作**，需在改进 delete 判断逻辑并提升 LLM 稳定性后再推进自动化。

---

## 二、链路各环节评估

### 2.1 数据筛选质量

#### 漏斗概况

原始对话数据经过两轮处理，最终保留率约为 33%：

```
原始对话       1514 条
  ↓ V2 Prompt + 人工抽样过滤（37%淘汰）
保留           951 条
  ↓ 过滤「模型回复解析失败」session（20%淘汰）
有效 session   761 条
  ↓ 转换为 scenario 格式（每条对话可产出多个场景）
scenario      1658 条
  ↓ 均匀抽样（A/B/C/D/E 各 100 条）
最终样本        500 条
```

#### 评估

| 维度 | 情况 | 风险等级 |
|------|------|----------|
| 样本代表性 | 5 类均匀抽样，分布可控 | 🟢 低 |
| 解析失败过滤 | 20% 因格式问题被丢弃 | 🟡 中 |
| 人工标注一致性 | 抽样审核，未做全量双标 | 🟡 中 |

**影响**：20% 的 session 因解析失败被直接过滤，意味着这类数据在真实场景中同样无法被正常处理。若生产数据中这类格式问题比例相近，系统实际覆盖率会比预期低。

**建议**：对「解析失败」的 session 单独分析失败原因，区分「模型输出格式问题」和「数据本身质量问题」，前者应作为提示词工程的改进输入。

### 2.2 记忆提取质量

#### 核心指标

- **提取覆盖率**：500 条 scenario 中，368 条有产出 → **73.6%**
- **无提取率**：132 条 scenario（26.4%）被 LLM 判断为无值得保存内容
- **JSON 解析失败率**：约 30 条（**6%**），主因是 MiniMax 输出偶发截断
- **平均产出密度**：每个有效 scenario 生成 **1.67 条** memory

#### Memory 类型分布

| 类型 | 数量 | 占比 | 说明 |
|------|------|------|------|
| user | 387 | 46.5% | 用户个人偏好/背景 |
| project | 193 | 23.2% | 项目相关上下文 |
| feedback | 146 | 17.5% | 用户对 AI 的反馈 |
| reference | 107 | 12.8% | 参考资料/知识点 |

#### Expected Action 分布

| 动作 | 数量 | 占比 |
|------|------|------|
| keep（保留） | 351 | 42.1% |
| delete（删除） | 295 | 35.4% |
| review（复查） | 187 | 22.5% |

**影响**：6% 的 JSON 解析失败意味着每 17 条提取的 memory 中，就有 1 条会因格式问题静默丢失，且当前没有重试或降级机制。26.4% 的无提取率需要结合人工抽样验证——部分「无值得保存内容」的判断可能是漏提取，而非真的没有价值。

**建议**：
1. 针对 JSON 截断问题，增加输出长度的截断检测与自动重试（最多 1 次）；
2. 对 132 条无提取 scenario 抽取 30 条人工复核，验证「无产出」是否符合预期。

### 2.3 评分引擎准确率

#### 混淆矩阵

下表展示 200 条有效评分中，Ground Truth 期望动作与引擎实际判断的对应关系：

|  | 预测 keep | 预测 review | 预测 delete |
|--|-----------|-------------|-------------|
| **期望 keep**（87条） | ✅ 62（71%） | 21（24%） | ❌ 4（5%） |
| **期望 delete**（73条） | ❌ 31（42%） | 33（45%） | ✅ 9（12%） |
| **期望 review**（40条） | 15（38%） | ✅ 18（45%） | 7（18%） |

#### 关键准确率指标

| 指标 | 数值 | 评价 |
|------|------|------|
| keep 类精确率 | 62/(62+4) = **94%** | 🟢 优秀 |
| delete 类精确率 | 9/(9+31) = **22%** | 🔴 严重不足 |
| review 类精确率 | 18/(18+7) = **72%** | 🟡 中等 |
| 整体正确率（严格） | (62+9+18)/200 = **44.5%** | 🔴 偏低 |
| 整体正确率（含偏保守可接受） | (62+9+18+21+33)/200 = **71.5%** | 🟡 可接受 |

> 注："偏保守可接受"指把应删除/应复查的内容判为 review 的情形——虽然不精确，但不会误删有价值内容。

#### 综合分分布

| 分段 | 数量 | 占比 |
|------|------|------|
| 5.0 分（满分） | 24 | 12% |
| 4.0–4.9 | 57 | 28.5% |
| 3.5–3.9（保留区间下沿） | 29 | 14.5% |
| 2.5–3.4（复查区间） | 80 | 40% |
| <2.5（删除） | 10 | 5% |

**关键观察**：复查区间（2.5–3.4）聚集了 40% 的评分结果，说明引擎对大量内容的判断处于「不确定」状态，倾向于推给人工复查，而非给出明确结论。

#### LLM 稳定性问题

42 条（**21%**）在 MiniMax 限额耗尽后返回异常，被系统默认标为 3.0 分（复查）。这批数据的真实评分未知，直接拉低了准确率统计的可信度——**上述所有准确率数字均建立在「仅剩 79% 有效评分」的基础上**。

## 三、核心问题深度分析

### 问题一：临时性内容被系统性误判为 keep（最严重）

**数据**：73 条期望 delete 的 memory 中，有 31 条（42%）被评为 keep，另有 33 条（45%）被评为 review。只有 9 条（12%）被正确判断为删除。

**根因：评分维度缺乏"任务性检测"**

当前 4 维评分（重要性、时效性、可信度、准确性）本质上是在评估内容质量，而非评估内容是否适合长期记忆。一段「写得很好的单次任务背景」，在这 4 个维度上都可能得高分，但它就是不应该被保留。

**典型案例 1 — story_background.md**（B 类，应删，实得 3.6 分被保留）

> 内容：同人文创作任务的角色背景设定
> 引擎理由：「包含完整的情节前提和角色关系铺垫，对写作有明确指导意义……属于稳定的创作背景信息，非临时任务状态」

这里暴露了一个认知混淆：引擎把「内容稳定」等同于「值得长期记忆」。但这段背景只服务于这一次创作任务，任务结束后毫无价值。**引擎缺少一个关键判断维度：这条 memory 是否只服务于一次性任务，而非用户的长期背景？**

**典型案例 2 — security_feedback_scam_awareness.md**（E 类，应删，实得 4.8 分被保留）

> 内容：AI 警告用户游戏交易中「定金骗局」「30 天冷却期」等诈骗手法
> 引擎理由：「诈骗手法稳定，来源清晰，记录准确。未来遇到类似交易场景时可帮助用户识别风险」

这是一个更严重的误判。引擎将「AI 说给用户听的内容」错误地保存为「用户自身的知识/偏好」。Memory 的本质是记录**用户是谁、用户想要什么**，而不是 AI 的输出内容。这条 memory 把 AI 的建议当成了用户的属性，从根本上就错了。

**影响**：delete 类精确率 22% 意味着引擎无法可靠地清理无用记忆，若长期运行，记忆库会持续膨胀，充斥大量无效内容，反而降低真正有价值信息的检索质量。

---

### 问题二：高价值情感/心理信号被过度删除（少但影响大）

**数据**：87 条期望 keep 的 memory 中，有 4 条被误判为 delete。数量少，但错误代价高——这 4 条都是用户的重要长期背景信息。

**典型案例 — upcoming_valentines_day_stress.md**（A 类，应保留，实得 4.0 分被删除）

> 内容：用户对节日约会有严重焦虑，与男友的约会安排让她感到窒息
> 引擎理由：「属于典型的临时状态记录：特定节日引发的短期情绪应激」

引擎的时效性规则在这里过度触发了。节日这个时间标签触发了「临时状态」的判断，但内容本身揭示的是用户有**社交焦虑和关系压力**的长期心理倾向——这正是高质量个性化服务最需要的背景信息。

**影响**：虽然只有 4 条，但这类误删是「不可逆的损失」——一旦删除，这些用户背景就永远消失了。误留一条无用内容代价小，误删一条关键背景代价大。两类错误的代价不对称，引擎当前的权重设置没有体现这一点。

---

### 问题三：21% 评分失败导致准确率统计可信度下降

**数据**：200 条有效评分实际上来自 242 条评分请求，其中 42 条（21%）因 MiniMax 限额耗尽而失败，被强制标为 3.0 分。

**影响**：
1. 准确率统计仅基于 79% 的数据，存在**选择性偏差**——失败的 42 条集中在 Benchmark 后期，可能存在分布不均
2. 在生产环境中，这 21% 的 memory 评分结论不可信，但系统仍然会据此执行保留/删除操作
3. 3.0 分的默认值落在复查区间，表面上「安全」，但掩盖了真实分布，可能有部分本应删除的内容被默默留存

## 四、改进建议

### 建议一：增加「任务性检测」维度（高优先级）

**目标问题**：临时任务背景被误判为 keep（31 条）

**具体方向**：在评分 Prompt 中增加第五个前置判断：

> 「这条 memory 记录的，是用户的长期背景/偏好/特征，还是单次任务的上下文？如果是单次任务，直接输出 delete，不进入后续评分。」

判断标准可包含：
- 内容中是否包含「帮我写」「这次任务」「本次对话」等任务性语言
- 内容是否描述了一个有明确开始/结束的事件，而非用户的持续属性
- 内容的主语是「AI 的建议/输出」还是「用户自身」

**预期效果**：若能正确拦截 B/E 类内容，delete 精确率应从 22% 提升至 50%+。

---

### 建议二：引入「误删代价权重」，对高价值内容降低删除阈值（中优先级）

**目标问题**：A 类情感/心理信号被过度删除（4 条）

**具体方向**：在最终决策逻辑中，当综合分在删除区间（<2.5）但内容类型为 `user`（用户个人属性）时，自动升级为 `review`，不直接删除。

逻辑如下：
```
if score < 2.5 and memory_type == "user":
    action = "review"  # 不直接删除，交给人工
else:
    action = "delete"
```

**预期效果**：在不影响 delete 精确率的前提下，消除高价值用户背景被误删的风险。

---

### 建议三：为 LLM 调用增加重试与降级机制（高优先级）

**目标问题**：21% 评分失败被默认标为 3.0 分

**具体方向**：
1. **重试**：检测到 LLM 返回异常时，自动重试 1 次（间隔 5 秒）
2. **降级**：重试仍失败时，标记为 `score_failed`，在 UI 上显著区别于正常复查，不混入常规 3.0 分池
3. **批次限流**：在 Benchmark 或批量评分场景中，增加 token 消耗监控，接近限额时主动暂停并报警，而非等到失败后被动处理

**预期效果**：将评分失败率从 21% 降至 <5%，显著提升准确率统计的可信度。

---

### 建议四：扩充 delete 类训练样本，优化评分 Prompt（中优先级）

**目标问题**：引擎对 delete 类内容的整体判断能力薄弱

**具体方向**：
1. 从本次 Benchmark 中提取 31 条「应 delete 但被误判为 keep」的案例，作为 Few-shot 反例加入评分 Prompt
2. 在 Prompt 中明确列举「应该删除的典型特征」，例如：
   - AI 给用户的单次建议（主语是 AI，不是用户）
   - 单次创作/写作任务的背景素材
   - 某个具体事件的细节描述，且该事件已结束
3. 每月更新一次 Few-shot 案例库，保持与真实数据分布同步

**预期效果**：配合建议一，综合 delete 精确率目标达到 60% 以上。

## 五、整体评级与结论

### 评级：C（有条件可用）

| 评级标准 | 说明 |
|----------|------|
| A | 各环节准确率达标，可全自动运行 |
| B | 主要环节可用，个别问题需要监控 |
| **C** | **核心环节存在结构性缺陷，需人工兜底，不建议全自动执行删除** |
| D | 系统性失效，不建议使用 |

### 评级依据

| 维度 | 得分 | 说明 |
|------|------|------|
| 记忆提取覆盖率 | 🟢 | 73.6%，基本符合预期 |
| 记忆提取稳定性 | 🟡 | 6% JSON 失败率，需改进但可接受 |
| keep 类判断准确率 | 🟢 | 94%，表现优秀 |
| delete 类判断准确率 | 🔴 | 22%，严重不足，是最大风险点 |
| LLM 稳定性 | 🔴 | 21% 评分失败，污染统计可信度 |
| 整体系统可信度 | 🟡 | 71.5%（宽松口径），但含大量「不确定」 |

### 最终结论

Memory Quality MCP 的**保留判断（keep）已可用**，可以信任其「这条值得保留」的结论。但**删除判断（delete）尚不可信**——当前仅有 22% 的删除建议是正确的，78% 的情况下它会放走本该清理的内容，或把本该保留的重要背景信息误删。

**建议的推进节奏**：

1. **近期（1-2周）**：修复 LLM 重试/降级机制，消除评分失败问题，获得真实准确率基线
2. **中期（2-4周）**：在评分 Prompt 中增加「任务性检测」维度，重新跑 Benchmark 验证改进效果
3. **上线前**：delete 类精确率须达到 **≥60%** 方可开启自动删除功能；在此之前，delete 动作全部降级为人工复查
