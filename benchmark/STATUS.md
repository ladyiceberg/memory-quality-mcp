### 评估报告状态

`EVALUATION_REPORT.md` 已完成，基于 **500 条 scenario、200 条有效评分** 的真实跑出来的数据。

结果**没有回写到 JSON 文件**——报告写完时是内存里跑完直接出报告，中间数据没持久化。所以文件是空的，但报告数字是真实的。

### 核心结论（来自报告）

评分引擎存在**结构性缺陷**：

- **delete 精确率只有 22%**：大量"应该删除"的临时性内容被误判为保留
- **LLM 调用失败率 21%**：失败的被强制打成 3.0 复查，污染准确率统计
- 记忆提取覆盖率 73.6%，JSON 解析失败率 6%，属于可接受范围

当前系统**不建议在生产环境全自动执行删除操作**。

---

## 决策

先把产品推出去收集真实用户反馈，不在评分引擎改进上投入时间。

Benchmark 当前的 baseline 数字（delete 精确率 22%）保留，等后续有时间时作为改进前的对比基准。

## 接下来要做的事（待做）

### 第一阶段：持久化评分结果（可选，低优先级）

把之前跑过的 500 条评分结果回写到 `scenarios.json`，方便后续复用，不用重跑。
- 需要重新跑一遍（约消耗 MiniMax 100 次调用左右）
- 或者重新设计脚本让每次评分实时写入文件，而不是最后汇总

### 第二阶段：改进评分引擎（核心，等用户反馈稳定后做）

目标：把 delete 精确率从 22% 提升到 60%+

方向：
1. **改进 Prompt**：当前对"临时性内容"的判断太弱，需要加强 `is_not_to_save` 的规则覆盖
2. **增加 Few-shot 示例**：在 prompt 里加几个典型的"应该删除"例子
3. **改进 fallback 逻辑**：LLM 失败时别强制打 3.0，改成重试一次或标记为 `uncertain`
4. **验证改进效果**：每次改完跑完整 500 条 benchmark 对比数字

### 第三阶段：human_scores 标注（配合第二阶段）

833 条 memory 的 `human_scores` 字段全为 null，需要人工标注才能做"LLM 评分 vs 人工评分"的对比。
- 工作量大，建议抽取 100-200 条做即可
- 优先标注 delete/keep 分歧最大的那批

## 触发条件

当出现以下任一情况时，回来推进上面的 TODO：

- 收到用户反馈"评分不准"的 issue 超过 3 条
- 产品有付费用户，需要提高可信度
- 有时间和 API 额度专门做一轮改进
