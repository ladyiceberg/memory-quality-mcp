# Memory Quality MCP · 设计方案 v2

> 目标：用一个 Claude MCP 插件验证「AI 记忆质量管理」这个机会点的真实需求
> 阶段：Demo 验证（不是生产级产品）
> 时间：6-8 周内跑通并发布
> 最后更新：2026-04-04

---

## 一、背景与核心假设

### 1.1 机会来源

本设计方案基于 2026-04-03 完成的综合调研（Reddit × HN × 竞品扫描 × GitHub 技术生态，调研文档见飞书）。

**核心发现**：AI 记忆生态存在清晰的分层空白：

```
应用层     ChatGPT Memory | Claude Projects（封闭黑盒，不开放 API）
              ↕
存储层     mem0 (41k★, $24M) | Zep | MemGPT（成熟，竞争激烈）
              ↕
⚠️ 质量/过滤层 ← 几乎没有专注于此的独立产品，这里是空白
              ↕
检索层     向量数据库 Pinecone/Qdrant（成熟基础设施）
```

mem0 有基础质量判断，但**硬编码、不可配置、不是独立模块**。这一层是本产品的切入点。

### 1.2 要验证的假设

| # | 假设 | 验证指标 |
|---|------|---------|
| H1 | Claude 用户感受到「记忆越积越乱」的痛点 | 安装量 > 50 |
| H2 | 用户愿意把「记忆质量审查」这件事交给工具做 | 7 日留存 > 30% |
| H3 | 「主动告诉用户该删什么」比「被动存储」更有价值 | 用户主动触发审查次数 |
| H4 | 开发者愿意为记忆质量 API 付费 | 收到 ≥5 个开发者主动询问 |

## 二、目标用户与平台策略

### 2.1 目标用户决策（显式说明）

调研中识别出两类潜在用户：

| | AI 应用开发者（mem0/LangChain 用户） | C 端 Power User（Claude 重度用户）|
|---|---|---|
| 付费意愿 | 高 | 中 |
| 反馈质量 | 高，能精确描述痛点 | 低，只知道好用/难用 |
| 冷启动难度 | 高（HN 信号弱，倾向自己实现） | 低（直接安装即用） |
| 用户量 | 小 | 大 |
| H4 验证 | ✅ 唯一能验证 | ❌ 无法验证 |

**选择**：以 **C 端 Claude Code 重度用户** 作为第一阶段目标用户。

**理由**：
- 调研最高频痛点（Reddit ⭐⭐⭐⭐）来自 C 端用户，措辞高度一致，真实性已交叉验证
- Claude Code 的 MCP memory tool 是开放接口，可以直接读写用户的原生记忆（不同于 ChatGPT Memory 黑盒）
- Claude Code 用户是重度使用者，记忆问题对他们影响最大，愿意提供有效反馈
- 直接面向用户，反馈循环最短，无需经过开发者集成这一环

**⚠️ 技术前提**：在第一行代码之前，必须确认 MCP memory tool 的实际能力边界：能读什么、能写什么、能否删除、覆盖哪个版本的 Claude。这是整个设计的地基。

### 2.2 两阶段平台策略

**阶段一（第 1-8 周）：Claude Code MCP 插件**
- 目标：验证评分模型有效性，收集 20-30 个真实用户的定性反馈
- 成功指标：评分建议的用户接受率 > 60%
- 重点：质量，不是用户量

**阶段二（第 9 周之后）：OpenClaw ClawHub 插件**
- OpenClaw：开源个人 AI 助手，1200 万用户，70% 中国用户，有独立插件体系（ClawHub）
- 注意：OpenClaw **不支持 MCP 协议**，需要单独适配 ClawHub 插件格式（两套独立开发）
- 前提条件：阶段一验证评分模型有效（接受率达标）后，带着被验证的产品去铺量
- 中国用户群体与我们的真实数据资产高度匹配（见第三节）

**不在第一阶段铺量的原因**：记忆管理工具的核心信任来自「它帮我删的东西，确实该删」。在评分模型未校准之前铺量，建议准确率不高会造成口碑损失，难以挽回。

## 三、差异化资产：真实用户数据

### 3.1 数据资产描述

我们持有一批来自真实 Chatbot 产品的脱敏用户对话数据。这是本产品相对于所有竞争者的核心差异化：

```
市面上所有「记忆质量」相关工作（学术 + 开源）
  使用的是：合成数据 / 学术对话数据集（研究员自己造的）

我们有的是：真实用户在真实场景里说过的真实的话
  → 包含各种合成数据造不出来的细腻模式：
    用户的随口一说、被误解的偏好、时效性失效的老记忆
```

### 3.2 数据用途

**用途一：评分模型校准（不是拍权重，是数据支撑的判断）**
- 从真实数据里提取候选记忆，人工标注质量分
- 验证：被评为高分的记忆，用户后来真的反复引用了吗？
- 将评分模型从「合理猜测」变成「数据支撑的判断」

**用途二：Benchmark 小数据集构造**（见第五节）

**用途三：发现意想不到的痛点模式**
- 真实数据可能揭示调研未覆盖的第四类、第五类痛点模式

### 3.3 数据使用策略（路径 B + C）

**路径 B：显性记忆信号**（优先，标注成本低）
- 找用户明确说了「记住这个」的片段 → 天然正例
- 找用户纠正 AI 记忆的片段（「你上次说错了」）→ 天然负例
- 这类信号最干净，不需要太多人工判断

**路径 C：「随口一说」负例构造**
- 找「说了但不该被记住的话」（临时状态、随口一句）
- 这类数据合成难度极高，真实数据里天然存在

## 四、产品定位

**一句话**：给 Claude 的记忆层加一个「体检」功能——告诉你现在记了什么、哪些该删、哪些冲突了、哪些记偏了。

**不做什么**：
- ❌ 不替代 Claude 的原生记忆（不改底层存储架构）
- ❌ 不做跨平台同步（ChatGPT/Gemini API 不开放，做不了）
- ❌ 不做复杂的记忆图谱（验证阶段不需要）

**做什么**：
- ✅ 读取用户当前的 Claude 记忆（通过 MCP memory tool 接口）
- ✅ 四维质量评分：重要性、时效性、可信度、准确性（见第五节）
- ✅ 冲突检测：发现语义矛盾的记忆对
- ✅ 过时识别：标记可能已失效的记忆
- ✅ 给出可执行建议：「这 3 条建议删除，原因是……」
- ✅ 支持用户确认后一键清理（默认预览模式）

## 五、质量评分模型

### 5.1 四维评分框架

每条记忆沿四个维度打分（1-5 分）：

| 维度 | 定义 | 判断依据 | 权重 |
|------|------|---------|------|
| **重要性** | 这条信息对未来对话有多大帮助 | 是否包含具体偏好/事实/决策；是否可重复使用 | 40% |
| **时效性** | 这条信息现在还准确吗 | 包含时间词（"现在"/"最近"）；可能变化的事实；距上次引用时长 | 25% |
| **可信度** | 这条记忆是否有明确来源 | 用户明确陈述 vs AI 自行推断；来源是否清晰可追溯 | 15% |
| **准确性** | 记录内容是否忠实于来源，有无偏差 | 仅在可信度 ≥3 时评估；检测 AI 提取时是否过度解读或产生偏误 | 20% |

**设计说明（可信度 vs 准确性 的区分）**：
- 可信度 = **来源是否存在且明确**（这条记忆有没有依据）
- 准确性 = **记录是否忠实于来源**（有依据的情况下，记没记对）
- 两者正交：高可信度（有明确来源）+ 低准确性（但记歪了）→ 这正是「记错了还不停用」痛点的根源
- 可信度 < 3 时，准确性标记为「无法评估」（没有来源无从比对）

**⚠️ 权重说明**：当前权重为初始设计值。将通过真实数据（路径 B+C）校准，最终权重以数据验证结果为准。

**综合分计算**：
```
综合分 = 重要性×40% + 时效性×25% + 可信度×15% + 准确性×20%
（准确性无法评估时，权重分摊至其余三维，比例不变）

综合分 < 2.5  → 建议删除
综合分 2.5-3.5 → 建议复查
综合分 > 3.5  → 保留
```

### 5.2 冲突检测

用 LLM 对所有记忆做语义矛盾检测：

```
例：
  记忆 A："用户喜欢简洁的代码风格"
  记忆 B："用户希望代码注释详尽"
  → 检测为潜在冲突，提示用户确认哪条更准确
```

实现：批量发送给 Claude，用结构化 prompt 输出冲突对列表。

### 5.3 过时识别

识别以下模式的记忆为「可能过时」：
- 包含明确时间词（"今年"、"最近"、"现在在做"）
- 包含职位/项目/地点等可变信息
- 距上次引用超过 90 天（如有时间戳）

## 六、核心工具（MCP Tools）

```
memory_audit()
  → 扫描所有项目记忆，运行规则引擎快速初筛（不调 LLM）
  → 返回：健康摘要（总数、过时数、索引使用率、预估 LLM 调用次数）

memory_report(verbose=False)
  → 完整四维评分 + 冲突检测（调 LLM）
  → 返回：结构化列表，含建议操作（保留/复查/删除）及原因
  → 结果自动缓存到 SQLite，供 cleanup/dashboard 复用

memory_cleanup(dry_run=True)
  → 优先读缓存执行清理，无缓存时重新评分
  → dry_run=True 时只预览不执行，永远不静默删除
  → 删除前自动备份到 .trash/<timestamp>/，同步更新 MEMORY.md 索引

memory_score(content: str)
  → 单条记忆四维打分（调试 + Benchmark 校验用）
  → 规则引擎命中时不调 LLM

memory_dashboard()
  → 从缓存生成苹果极简风 HTML 健康报告，用系统默认浏览器打开
  → 包含健康分圆环、汇总统计、可折叠记忆清单
  → 无需重复调 LLM，复用 memory_report() 缓存
```

## 七、技术架构

```
用户 Claude Code 对话
      │
      ▼
MCP Server（本地 Python 进程）
      │
      ├── memory_reader.py  ← 扫描 ~/.claude/projects/*/memory/，读 frontmatter
      │
      ├── quality_engine.py ← 规则引擎（零成本初筛）+ LLM 四维评分 + 冲突检测
      │       └── llm_client.py（统一多提供商接口：OpenAI/Kimi/MiniMax/Anthropic）
      │
      ├── session_store.py  ← SQLite 持久化 report 结果，供 cleanup/dashboard 复用
      │
      ├── memory_writer.py  ← 安全删除 + .trash 备份 + MEMORY.md 索引同步
      │
      └── dashboard.py      ← 生成苹果极简风 HTML 报告，webbrowser.open() 打开
```

**技术选型**：

| 组件 | 选型 | 理由 |
|------|------|------|
| MCP 框架 | Python MCP SDK（官方） | 最稳定，文档最全 |
| LLM 调用 | 统一 OpenAI 兼容接口 | 支持 OpenAI/Kimi/MiniMax/Anthropic，用户自由选择 |
| 记忆读取 | 直接读写本地 .md 文件 | Claude Code Auto Memory 存在 ~/.claude/projects/*/memory/ |
| 会话状态 | 本地 SQLite（~/.memory-quality-mcp/session.db） | report 结果跨工具调用复用，cleanup 无需重复调 LLM |
| 可视化 | 纯 HTML+CSS+内联 JS | 零外部依赖，webbrowser.open() 打开，苹果极简风 |
| 评分校准 | 真实用户 Chatbot 脱敏数据 | 核心差异化资产，权重数据支撑而非拍板 |
| 配置 | ~/.memory-quality-mcp/config.yaml | 用户级配置，打包安装后仍可修改 |

## 八、Benchmark 小数据集

### 8.1 定位与目标

- **不是**构建完整的记忆质量评估体系
- **是**：100 条人工标注的记忆样本，每条附带四维人工评分
- **价值**：
  1. 内部：持续衡量评分模型的准确率（模型分 vs 人工分的相关性）
  2. 外部：开源后占住「记忆质量评估」这一子领域的第一个数据集位置

### 8.2 构造方法

**正例来源（路径 B）**：
- 从真实 Chatbot 数据里找用户明确确认的记忆（「记住这个」「对，就是这样」）
- 天然高质量标注，无需过多人工介入

**负例来源（路径 C）**：
- 「随口一说」型：临时状态、情绪化表达被错误记录
- 「记偏了」型：AI 提取时过度解读用户意图
- 「过时了」型：含时间词的旧状态描述

**标注标准**：四个维度各独立打分（1-5），标注人先独立打分再对齐，分歧 > 1 分的条目进入复议。

### 8.3 与现有 Benchmark 的关系

现有记忆 Benchmark（LongMemEval、MemoryAgentBench、LOCOMO 等）测的是「召回准确率」，不是「质量判断」，无法直接复用。冲突检测子任务可借用 NLI 数据集（SNLI/MultiNLI）里的矛盾句对作为测试样本。

## 九、用户体验设计

### 9.1 安装方式

```json
{
  "mcpServers": {
    "memory-quality": {
      "command": "uvx",
      "args": ["memory-quality-mcp"]
    }
  }
}
```

目标：安装时间 < 2 分钟，无需注册账号。

### 9.2 典型对话流程

```
用户：帮我检查一下我的记忆质量

Claude：[调用 memory_audit()]
        好的，我帮你扫描了一下。你目前有 47 条记忆：
        - 🔴 3 对存在冲突（例如：你同时记了「喜欢早起」和「习惯晚睡」）
        - 🟠 2 条可能记偏了（AI 提取时可能过度解读了你的原意）
        - 🟡 8 条可能已过时（含「最近」「现在」等时间词）
        - ⚪ 12 条重要性偏低（如临时状态、随口一说）

        要我列出建议清理的清单吗？

用户：列出来

Claude：[调用 memory_report()]
        建议处理的 15 条记忆：

        ⚠️ 可能记偏了（需你确认）：
        1. 原文："用户说想早点睡" → 记录为："用户偏好早睡习惯"
           (准确性 1/5：一次随口的话被固化为长期习惯，建议复查)

        🗑 建议删除：
        2. "用户在 2023 年做的项目是 X"（时效性 1/5，可能已过时）
        ...

        确认清理这 15 条吗？（我不会动其他记忆）

用户：确认

Claude：[调用 memory_cleanup()]
        ✅ 已清理 15 条，剩余 32 条高质量记忆。
```

## 十、验证计划

### 阶段一：构建（第 1-3 周）

- [ ] 本地测试通过，能跑完完整对话流程

### 阶段二：发布（第 4 周）

- [ ] 发布到 PyPI（`pip install memory-quality-mcp`）
- [ ] 提交到 mcp.so 插件目录
- [ ] 在 r/ClaudeAI 发一篇使用体验帖（真实使用记录，不是广告）
- [ ] 在 Claude Discord 的 #tools 频道分享

### 阶段三：观察与校准（第 5-8 周）

每周记录：
- 安装量（PyPI download stats）
- **评分建议接受率**（用户确认清理 vs 跳过的比例）← 核心指标
- GitHub star 数
- 用户 issue/反馈

**决策节点（第 8 周）**：
- 安装量 > 50 且接受率 > 60% → 评分模型有效，准备阶段二 OpenClaw 扩展

- 安装量 < 20 → 调整渠道或重新考虑产品方向
- 收到 ≥5 个开发者询问 API → 评估是否转向 B2B SDK

### 阶段四：OpenClaw 扩展（第 9 周之后，条件触发）

- [ ] 触发条件：阶段一接受率 > 60%
- [ ] 适配 OpenClaw ClawHub 插件格式（独立开发，不复用 MCP 代码）
- [ ] 目标用户：中国 C 端用户（与真实数据人群高度匹配）

## 十一、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| MCP memory tool 读写权限受限 | 中 | Week 0 优先确认；若只读则先用 mock 数据验证 UX，调整 cleanup() 为「建议清单导出」形态 |

| OpenClaw 插件生态快速壮大，质量管理出现竞争者 | 中 | 阶段一快速占位，建立第一批用户口碑 |

## 附录：与 v1 设计方案的核心变更

| 变更点 | v1 | v2 |
|--------|----|----|
| 目标用户 | 未显式说明 | 明确为 Claude Code C 端用户，原因已论证 |
| 平台策略 | 只提 Claude MCP | 两阶段：Claude Code 先验证，OpenClaw 后铺量 |
| 评分维度 | 三维（重要性/时效性/可信度） | 四维（新增准确性，区分来源有无 vs 记录是否忠实）|
| 数据资产 | 未提及 | 核心差异化，真实 Chatbot 数据用于模型校准和 Benchmark 构造 |
| Benchmark | 未提及 | 标注样本，真实数据路径 B+C 构造，有具体方法论 |
| 验证计划 | 构建→发布→观察 | 新增 Week 0 技术前提核实 + Benchmark 构造；核心指标改为「接受率」|
| 对话示例 | 无「记偏了」场景 | 新增「记偏了」场景，对应最高频 Reddit 痛点 |
| LLM 支持 | 仅 Anthropic | 多提供商（OpenAI/Kimi/MiniMax/Anthropic），OpenAI 兼容接口统一调用 |
| 会话状态 | 无 | SQLite 持久化 report 结果，cleanup/dashboard 复用，不重复调 LLM |
| 可视化 | 无 | memory_dashboard()：苹果极简风 HTML 报告，浏览器打开 |
| 配置路径 | 项目目录 config.yaml | ~/.memory-quality-mcp/config.yaml，打包后用户可修改 |
