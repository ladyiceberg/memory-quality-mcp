# Memory Quality MCP · 设计方案

> 目标：用一个 Claude MCP 插件验证「AI 记忆质量管理」这个机会点的真实需求
> 阶段：Demo 验证（不是生产级产品）
> 时间：6-8 周内跑通并发布

---

## 一、核心假设（要验证的问题）

| # | 假设 | 验证指标 |
|---|------|---------|
| H1 | Claude 用户感受到「记忆越积越乱」的痛点 | 安装量 > 50 |
| H2 | 用户愿意把「记忆质量审查」这件事交给工具做 | 7日留存 > 30% |
| H3 | 「主动告诉用户该删什么」比「被动存储」更有价值 | 用户主动触发审查次数 |
| H4 | 开发者愿意为记忆质量 API 付费 | 收到 ≥5 个开发者主动询问 |

---

## 二、产品定位

**一句话**：给 Claude 的记忆层加一个「体检」功能——告诉你现在记了什么、哪些该删、哪些冲突了。

**不做什么**：
- ❌ 不替代 Claude 的原生记忆（不改底层存储）
- ❌ 不做跨平台同步（ChatGPT/Gemini API 不开放，做不了）
- ❌ 不做复杂的记忆图谱（验证阶段不需要）

**做什么**：
- ✅ 读取用户当前的 Claude 记忆（通过 MCP memory tool 接口）
- ✅ 分析记忆质量：重要性评分、冲突检测、过时识别
- ✅ 给出可执行建议：「这 3 条建议删除，原因是……」
- ✅ 支持用户确认后一键清理

## 三、功能设计

### 3.1 核心工具（MCP Tools）

MCP 插件向 Claude 暴露以下工具：

```
memory_audit()
  → 读取当前所有记忆，运行质量分析
  → 返回：健康报告（总数、冲突数、过时数、建议清理数）

memory_report(verbose=False)
  → 输出详细的记忆清单，每条附带质量评分
  → 返回：结构化列表，含建议操作（保留/更新/删除）

memory_cleanup(dry_run=True)
  → 执行清理：删除低质量记忆，合并冲突条目
  → dry_run=True 时只预览不执行，保护用户数据

memory_score(content: str)
  → 单条记忆质量打分（开发者调试用）
  → 返回：重要性/时效性/可信度三维评分
```

### 3.2 质量评分模型

每条记忆沿三个维度打分（1-5分）：

| 维度 | 含义 | 判断依据 |
|------|------|---------|
| **重要性** | 这条信息对未来对话有多大帮助 | 是否包含具体偏好/事实/决策；通用性/可重复使用 |
| **时效性** | 这条信息现在还准确吗 | 包含时间词（"现在"/"最近"）、可能变化的事实 |
| **可信度** | 这条信息来源是否可靠 | 是否来自用户明确陈述 vs 推断 |

**综合分** = 重要性×50% + 时效性×30% + 可信度×20%

综合分 < 2.5 → 建议删除
综合分 2.5-3.5 → 建议复查
综合分 > 3.5 → 保留

### 3.3 冲突检测

用 LLM 对所有记忆两两比较语义矛盾：

```
例：
  记忆A："用户喜欢简洁的代码风格"
  记忆B："用户希望代码注释详尽"
  → 检测为潜在冲突，提示用户确认
```

实现：批量发送给 Claude，用结构化 prompt 输出冲突对列表。

### 3.4 过时识别

识别以下模式的记忆为「可能过时」：
- 包含明确时间词（"今年"、"最近"、"现在在做"）
- 包含职位/项目/地点等可变信息
- 距上次引用超过 90 天（如有时间戳）

## 四、技术架构

```
用户 Claude 对话
      │
      ▼
MCP Server（本地 Python 进程）
      │
      ├── memory_read()    ← 通过 Claude MCP memory tool 读取记忆
      │
      ├── quality_engine() ← 调用 LLM API 做质量评估
      │       └── 用 Claude claude-haiku-3-5（低成本）批量处理
      │
      └── memory_write()   ← 用户确认后执行清理操作
```

**技术选型**：

| 组件 | 选型 | 理由 |
|------|------|------|
| MCP 框架 | Python MCP SDK（官方） | 最稳定，文档最全 |
| LLM 调用 | Anthropic API（claude-haiku-3-5） | 成本低（$0.25/M tokens），速度快 |
| 记忆读取 | Claude MCP memory tool | 官方接口，无需 hack |
| 数据存储 | 本地 SQLite | 记录审查历史，方便用户回溯 |
| 配置 | 单一 config.yaml | 阈值、API key 等用户可调 |

**不依赖**：不需要向量数据库、不需要服务器、本地运行即可。

## 五、用户体验设计

### 5.1 安装方式

```bash
# 用户只需在 Claude Desktop 的 MCP 配置里加一行：
{
  "mcpServers": {
    "memory-quality": {
      "command": "uvx",
      "args": ["memory-quality-mcp"]
    }
  }
}
```

目标：安装时间 < 2 分钟，无需注册账号。

### 5.2 典型对话流程

```
用户：帮我检查一下我的记忆质量

Claude：[调用 memory_audit()]
        好的，我帮你扫描了一下。你目前有 47 条记忆：
        - 🟡 8 条可能已过时（含「最近」「现在」等时间词）
        - 🔴 3 对存在冲突（例如：你同时记了「喜欢早起」和「习惯晚睡」）
        - ⚪ 12 条重要性偏低（如：「用户说了一句好的」）

        要我列出建议删除的清单吗？

用户：列出来

Claude：[调用 memory_report()]
        建议删除的 12 条记忆：
        1. "用户在 2023 年做的项目是 X"（时效性 1/5，可能已过时）
        2. "用户说今天想早点睡"（重要性 1/5，临时状态不值得记）
        ...

        确认清理这 12 条吗？（我不会动其他记忆）

用户：确认

Claude：[调用 memory_cleanup()]
        ✅ 已清理 12 条，剩余 35 条高质量记忆。
```

## 七、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| Claude MCP memory tool 读取权限受限 | 中 | 先用本地 mock 数据验证 UX，上线后再测真实接口 |
| 用户觉得「AI 帮我删记忆」不安全 | 高 | 默认 dry_run=True，所有操作先预览后确认，永远不静默删除 |
| LLM 评分准确率不稳定 | 中 | 评分只做参考，最终决策权留给用户；同时收集用户对建议的接受率作为反馈 |
| 发布后无人关注 | 中 | 准备 3 个渠道（mcp.so / Reddit / Discord）同时分发，降低单渠道依赖 |
